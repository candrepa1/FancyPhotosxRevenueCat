Challenge 2

Dynamic subscription management

When your AI resources are under heavy load, you'll be able to disable the higher subscription tier remotely — no app update required.

Since the higher subscription tier should be available only at certain times, RevenueCat's scheduling target feature is the perfect fit for your use case.

Targeting by placement with scheduling

We can use placements to define specific rules for specific "places" in your app. We'll use it as a way to identify the AI features section, and target this placement to control which offering to show.

else if let offering = Purchases.shared.offerings()?.currentOffering(forPlacement: "ai-tab") {
    PaywallView(offering: offering)
}

We target "any audience" — every user in your app. The breakdown for the "ai-tab" placement:

  • Users who are entitled to the AI feature tier already → get access to features
  • Users who are entitled to other tiers → display paywall for AI feature tier
  • Users who haven't signed up to any tier → display paywall for AI feature tier

Then we define that we want to show the "high-power-ai-offering" for the "ai-tab" placement.

We also need to define the fallback for all other parts of the app that do not fall within the defined placement. In this case, these are all tiers available below the higher AI subscription tier.

  • When schedule is inactive → placement returns no offering → app shows "Currently unavailable" instead of the paywall.

FAQ

What happens to existing subscribers when the schedule expires?

They keep access — their entitlement stays active regardless of the schedule.

Can this be toggled manually outside the schedule?

Yes, from the RevenueCat dashboard at any time.

How quickly does the change propagate?

Next time the SDK fetches offerings — typically on app foreground.

Challenge 3

Fraud Prevention

Given the cost of your AI compute, here's how we ensure only paying customers get access to those features.

Since users who gain access to the high AI subscription tier are going to be using expensive compute, RevenueCat offers several features to prevent fraud:

1. Receipt Validation

It is one of RevenueCat's implementation responsibilities to verify receipts directly with the app stores, assuring transaction integrity. This comes baked into RevenueCat's backend so there's no action needed from your side.

2. Server-Side Feature Gating

We can check for the user's entitlements on the client, but there's always a risk of MiTM attacks which can bypass these checks. In order to add an extra protective layer, we recommend adding another check to the server providing your expensive AI compute.

You can use RevenueCat's REST API to check for a user's active entitlements and grant access that way.

3. Trusted Entitlements

MiTM attacks can tamper with request content within the communication with RevenueCat. For that, RevenueCat has built Trusted Entitlements which helps verify this communication.

After the verification, RevenueCat provides a verificationResult which returns multiple cases that you can then use to make your access logic more robust.

4. Refund Handling

For iOS, you can set up a RevenueCat configuration to handle refund requests automatically.

Given the cost of the AI features, we'd suggest starting with "Always prefer declining refunds" and adjusting based on your refund patterns over time.

RevenueCat will automatically send the user's consumption data to Apple directly in order to help them make informed refund decisions.
